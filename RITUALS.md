# 작업 루틴 (Rituals)

이 문서는 사용자가 Claude Code에게 내릴 **표준 트리거 명령어** 모음입니다.
사용자가 아래 명령어 중 하나를 입력하면, 해당 루틴을 수행하세요.

명령어 목록 한눈에 보기:
- `/시작` — 작업 시작 루틴
- `/종료` — 작업 종료 루틴
- `/완료` — 단위 작업 완료
- `/막힘` — 막혔을 때
- `/결정 [내용]` — 방향 전환 기록
- `/주간정리` — 주간 회고
- `/상태` — 빠른 3줄 요약

---

## 🌅 `/시작` — 작업 시작 루틴

사용자가 "/시작" 또는 "작업 시작"이라고 하면 다음을 수행:

1. **CLAUDE.md 읽기** — 진행 상황 체크리스트, 다음 작업 우선순위, Decision
   Log를 우선적으로 본다.
2. **현재 진행 상황을 5줄 이내로 요약** (완료된 것 / 진행 중인 것 / 다음 할 것).
3. **폴더 구조 훑기** — 문서(CLAUDE.md의 아키텍처 섹션)와 실제 코드/폴더
   구조 사이에 불일치가 있으면 그 차이를 보고한다.
4. **오늘 작업 후보 3개 추천** — 각각 예상 소요시간, 난이도(쉬움/보통/어려움)
   를 함께 제시.
5. **사용자의 선택을 기다림** — **먼저 코딩을 시작하지 말 것.** 사용자가
   고르거나 다른 지시를 내릴 때까지 대기.

---

## 🌙 `/종료` — 작업 종료 루틴

사용자가 "/종료" 또는 "작업 종료"라고 하면 다음을 수행:

1. **CLAUDE.md 업데이트**
   - 진행 상황 체크리스트: 완료 → `✅` (또는 `[x]`), 진행 중 → `🚧`
     (어디까지 했는지 한 줄 메모 포함).
   - 오늘 새로 발견된 작업 항목이 있으면 적절한 Phase에 추가.
2. **Decision Log 업데이트**
   - 오늘의 기술적 결정과 이유 (날짜 포함, YYYY-MM-DD 형식).
   - 시도했으나 실패한 접근법도 기록 (왜 실패했는지 함께).
3. **다음 작업 우선순위 갱신**
   - 다음 채팅이 바로 시작할 1순위 작업을 명시.
   - 그 작업의 컨텍스트와 주의사항(예: 의존성, 환경 변수, 미완 부분).
4. **오늘의 요약 보고** (사용자에게 출력)
   - 한 줄 요약
   - 변경된 파일 목록
   - 남은 이슈/버그
   - 개선 여지 있는 부분 (솔직하게)
5. 모든 업데이트 완료 후 사용자에게 보고.
   - 문서 변경은 **사용자가 어떤 부분이 어떻게 바뀌었는지 알 수 있도록**
     diff 요약을 함께 보여줄 것.

---

## ✅ `/완료` — 단위 작업 완료 루틴

기능 하나 또는 작업 단위가 끝났을 때 "/완료"라고 하면:

1. **CLAUDE.md 체크리스트의 해당 항목을 ✅로 변경**. 어떤 항목인지 사용자에게
   확인받고 진행 (잘못된 항목을 체크하지 않도록).
2. **새로 생긴 파일/함수가 있으면 아키텍처 섹션 업데이트** — 폴더 트리에
   파일 추가, 짧은 역할 설명.
3. **한 줄로 무엇이 완료됐는지 보고.**
4. **다음 작업 추천 1~2개** — 체크리스트의 다음 항목 또는 자연스럽게
   이어지는 작업.

---

## 🚧 `/막힘` — 막혔을 때

"/막힘" 또는 "/blocked"라고 하면:

1. **지금 시도하던 것과 왜 안 됐는지 Decision Log에 임시 기록** ("진행 중
   막힘:" 같은 마커로 구분). 나중에 해결되면 정식 결정으로 승격하거나 제거.
2. **다른 접근법 3개 제안** — 각각 장단점 함께. 가능하면 난이도와 예상
   소요시간도.
3. **사용자에게 선택권을 줌.** 임의로 한 가지를 골라 진행하지 말 것.

---

## 🔀 `/결정` — 방향 전환 시

"/결정 [내용]" 형식으로 입력하면:

1. **Decision Log에 결정 사항 + 이유 + 날짜 기록**. 입력된 내용을 그대로
   적지 말고, 이유를 함께 정리해 한 항목으로 만든다.
2. **영향받는 다른 문서를 자동 업데이트** — SPEC.md, README.md, CLAUDE.md
   중 해당 결정으로 내용이 바뀌어야 하는 부분을 찾아 갱신. 항상 **변경
   diff 요약을 사용자에게 보여주고 확인을 받은 뒤** 적용.
3. **영향받는 기존 코드 목록 보고** — grep으로 찾아 파일 경로 + 라인 번호로
   나열. 자동 수정은 하지 말고 사용자 지시를 기다린다.

---

## 📅 `/주간정리` — 주간 정리

일주일에 한 번 "/주간정리"라고 하면:

1. **CLAUDE.md 전체를 읽고 오래되거나 모순된 내용을 정리** — 변경 diff를
   사용자에게 보여주고 확인 후 적용.
2. **완료된 체크리스트 → "완료 이력" 섹션으로 이동** (Phase별 묶음을 유지).
3. **Decision Log 중 무효화된 결정을 표시** — 삭제하지 말고 `~~취소선~~`
   처리하고 무효화 사유와 날짜를 옆에 적는다.
4. **SPEC.md와 실제 구현 간 차이점 보고** — 둘 중 어느 쪽을 진실로
   삼을지는 사용자에게 묻는다.
5. **이번 주 성과 한 단락 회고** — 객관적 수치(추가된 파일 수, 끝낸 항목
   수)와 함께.

---

## 📊 `/상태` — 빠른 상황 체크

"/상태"라고 하면 3줄로 요약:
- 지금까지 완료한 것
- 지금 하고 있는 것
- 다음 할 것

**업데이트는 하지 말고 보고만** 한다. 문서 수정 없음.

---

## ⚠️ 루틴 사용 시 주의사항

- 루틴 명령어가 들어오면 다른 작업보다 **이걸 먼저** 수행한다.
- 루틴 도중 사용자가 다른 질문을 하면, 일단 그 질문에 답하고 루틴을 재개할지
  사용자에게 확인한다.
- **CLAUDE.md 업데이트는 항상 차이점을 보여주고 확인 받기.** 사용자가
  모르는 사이 문서가 바뀌면 안 된다. 작은 오타 수정도 마찬가지로 알린다.
- 루틴은 코드 작성과 별개의 트랙이다. 루틴 중에는 가급적 코드 변경을 하지
  않는다 (체크리스트/문서만 갱신). 코드 변경이 필요해 보이면 다음 작업
  후보로 제안한다.
- 모든 날짜는 ISO 형식(YYYY-MM-DD)으로 기록한다.
